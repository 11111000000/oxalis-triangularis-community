## Коротко

Мы создаём не «платформу ради платформы», а сад знаний:

- **растения важнее интерфейса** — сайт должен помогать, а не требовать обслуживания;
- **истина рождается между наблюдением и сомнением** — каждая запись имеет источники, статус уверенности и историю обсуждения;
- **модерация мягкая, но ясная** — правила минимальны, применяются последовательно;
- **инфраструктура течёт как вода** — статический сайт, GitHub PR/Issues, внешнее хранение медиа, минимум серверов;
- **агенты помогают, но не властвуют** — AI-агенты готовят черновики, проверки и ревью, решения принимают люди.

## Диалектическое основание

Тезис: сообществу нужен удобный сайт и живое участие множества людей.  
Антитезис: чем больше функций и инфраструктуры, тем выше стоимость поддержки и риск утонуть в администрировании.  
Синтез: статический публичный сайт + GitHub как источник истины + простая форма/Issue/PR-поток + ручная модерация с автоматическими помощниками.

Мы строим такие правила и технологии, которые позволяют людям вносить знания, не создавая лишней тяжести: хорошее управление заметно прежде всего по ясности процесса и низкой нагрузке на участников.

## Технический выбор

MVP сознательно простой:

1. **Статический сайт** в `site/`: HTML/CSS, без сборщика и node_modules.
2. **GitHub repository** как источник истины: записи хранятся в данных, изменения проходят через Issue/PR.
3. **Медиа вне Git**: фото лежат в Cloudinary/S3/GitHub Releases/другом хранилище, в записях только URL, автор и лицензия.
4. **Модерация через очередь**: сначала GitHub Issues/PR; позже можно добавить форму, которая автоматически создаёт Issue/PR.
5. **AI-агенты**: помогают проверить схему, найти дубли, улучшить текст, но не публикуют без человеческого review.

## Быстрый старт

### Вариант A: без Nix

```bash
npm run check
npm run serve
```

Откройте: `http://localhost:4173`.

Для шаблона подачи: `http://localhost:4173/submit.html`.

### Вариант B: dev shell

```bash
direnv allow
nix develop
just help
```

Полезные команды:

- `just check` — проверить проект
- `just serve` — запустить локальный сервер
- `just status` — посмотреть состояние git
- `just format` — отформатировать `flake.nix`
- `just dev` — подсказка по короткому старту

Скрипты не требуют установки зависимостей, а dev shell нужен только для удобства локальной работы.

## Правило простоты

Перед добавлением новой технологии задаём три вопроса:

1. Уменьшит ли это нагрузку модераторов через месяц?
2. Сможет ли новый участник понять это за 10 минут?
3. Можно ли добиться 80% пользы через GitHub/статический сайт/существующий сервис?

Если ответ «нет» — не добавляем.

## Следующий практический шаг

После локальной проверки можно открыть первый набор демонстрационных записей, сверить схему данных на реальных примерах и оформить шаблоны Issue/PR для участников. Ближайшие этапы перечислены в [Roadmap](ROADMAP.md).

Для работы в окружении используйте `direnv`, `nix develop` и `just`.
