# Governance: мягкий порядок

## Цель управления

Управление существует не для контроля ради контроля, а чтобы:

- защищать качество реестра;
- снижать нагрузку на модераторов;
- делать правила понятными новичку;
- сохранять доверие и доброжелательность.

## Роли

### Участник

Может предложить новую запись, исправление, фото, источник или вопрос. Участнику не нужно знать Git: достаточно формы или GitHub Issue.

### Доверенный участник

Получает статус после нескольких качественных вкладов. Может готовить PR с меньшим количеством ручных уточнений, но всё ещё проходит review.

### Модератор

Проверяет:

- соответствие схеме;
- источники;
- авторство и лицензию фотографий;
- отсутствие дублей;
- корректность тона обсуждения.

### Куратор данных

Отвечает за качество схемы, статусы записей и спорные таксономические случаи.

### Технический хранитель

Поддерживает сайт, CI, форму подачи и интеграции, но не усложняет архитектуру без необходимости.

## Принятие решений

1. Малые изменения — обычный PR + 1 review.
2. Изменения правил, схемы или архитектуры — Issue/RFC + обсуждение минимум 3 дня.
3. Спорные вопросы по сортам — статус `uncertain`, пока не появятся достаточные источники.
4. Невозвратные действия (удаление истории, force-push, массовые миграции) — только после явного согласия хранителей.

## Trust levels

- `new`: новые участники, все вклады проходят полную проверку.
- `steady`: несколько принятых вкладов, меньше уточнений.
- `trusted`: регулярные качественные вклады, может помогать в триаже.
- `curator`: назначается хранителями для предметной области.

Повышение статуса — не награда за активность, а знак предсказуемого качества.

## Как не утонуть

- Не добавлять отдельную БД, пока GitHub Issue/PR достаточно.
- Не хранить медиа в Git.
- Не создавать админку раньше, чем появится реальная очередь модерации.
- Не автоматизировать то, что меняется каждую неделю.
- Не спорить о совершенстве схемы, пока нет реальных данных.

## Ритм сообщества

Еженедельно:

- разбор новых заявок;
- закрытие очевидных дублей;
- публикация принятых записей.

Ежемесячно:

- ревизия спорных записей;
- улучшение правил;
- отчёт о состоянии реестра.

Ежеквартально:

- проверка, не стала ли инфраструктура тяжелее самого сада.
